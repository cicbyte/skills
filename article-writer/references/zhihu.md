## 平台定位

知乎用户期望**有深度、有思考**的技术内容。读者多为行业从业者和学习者，愿意阅读 3000 字以上的长文。SEO 权重高，搜索排名靠前。

## 标题风格

- 偏问题式或观点式：「如何理解……」「……是什么体验」「……的最佳实践」
- 标题可较长（15-30 字），但要精准切中读者搜索意图
- 避免标题党，知乎社区反感过度夸张的标题
- 好标题示例：
  - 「作为一个架构师，我是如何设计高并发系统的？」
  - 「2025 年了，Go 和 Rust 到底该怎么选？」
  - 「从零实现一个 Git：深入理解版本控制原理」

## 文章结构

### 标准结构（深度文章）

```
开头钩子（100-200字）
  → 抛出问题/分享经历/引用数据，快速建立专业形象

正文主体（2000-8000字）
  → 分层论述，每层有明确的小标题
  → 问题 → 分析 → 方案 → 对比 → 总结

结尾（200-500字）
  → 总结核心观点
  → 个人经验补充
  → 引发讨论的问题
```

### 回答式结构（回答问题时）

```
直接给出结论/答案（1-2句话）
  → 然后展开详细论述
  → 层层递进，先易后难
  → 最后总结 + 扩展阅读推荐
```

## 语言风格

- 正式但不刻板，保持**专业性和可读性的平衡**
- 用通俗易懂的语言解释复杂概念
- 适当加入个人观点和实战经验
- 善用类比帮助理解：「可以把消息队列理解为……」
- 避免纯术语堆砌，每个专业术语首次出现时给出简短解释
- 可以有适当的幽默，但不过度

## 排版规范

- 原生支持 Markdown，代码块渲染良好
- 支持数学公式（LaTeX 语法）
- 合理使用三级标题层级：H2 > H3 > H4
- 重要内容用**加粗**或> 引用块突出
- 每段 3-5 行，避免大段文字墙
- 善用分隔线 `---` 划分大章节
- 表格用于对比展示
- 图片配合说明文字

## 技术元素处理

### 代码
- 完整代码块，标注语言类型：```go / ```python
- 关键代码行加注释说明
- 超过 50 行的代码考虑折叠或拆分
- 附上运行结果或输出示例

### 仓库链接
- 直接内嵌 Markdown 链接
- 附上仓库简介：star 数、主要功能、适用场景
- 摘录 README 的核心内容

### 架构图
- 文字描述 + ASCII/Mermaid 图
- 配合表格对比各方案优劣

## 高赞技巧

1. **开头必须有钩子**：用数据/故事/争议性观点抓住注意力
2. **层次分明**：先说"是什么"再说"怎么做"最后说"为什么"
3. **有理有据**：引用官方文档、权威数据、个人实测数据
4. **对比分析**：展示多种方案的优劣，而不是只推一种
5. **实战导向**：附上可直接复用的代码/配置/命令
6. **持续更新**：标注文章更新时间，保持内容时效性
7. **引导互动**：结尾抛出开放性问题
